# Simplicity Pressure — Simplification Skill

The active framework for achieving real simplicity — not by dumbing down, but by understanding
deeply enough that the unnecessary becomes obvious.

**Persona context → read `../../SOUL.md`**
**Knowledge context → read `../../MEMORY.md`**

## The Core Principle

"Simple can be harder than complex: you have to work hard to get your thinking clean to make
it simple. But it's worth it in the end because once you get there, you can move mountains."

Simplicity is not:
- Removing features until nothing is left
- Making things easier to use by making them less powerful
- Hiding complexity behind one button (that's deception, not simplicity)

Simplicity is:
- Understanding the underlying problem so well that the inelegant solutions fall away naturally
- Having one concept that does what five used to do — because you understand both well enough
- Making the invisible architecture obvious through the visible design

## The Simplicity Audit

Run this on any product, feature set, or design:

### Step 1: List everything

Write down every element, feature, option, screen, button, configuration, or concept.
Don't filter. Everything goes on the list.

### Step 2: The first cut — ask "Why does this exist?"

For each item: why is it here? Who needs it? What breaks if it's gone?

- If the answer is "because users asked for it" → flag; does it serve the core experience?
- If the answer is "because a competitor has it" → cut; competitor-matching is not a reason
- If the answer is "because it was easier than solving the underlying problem" → cut; this is
  complexity disguised as a feature
- If no one can answer clearly → cut immediately

### Step 3: The second cut — the core experience question

What is the *one thing* this product must do brilliantly? *(From the product vision — see
`skills/product-vision`)*

For each remaining item: does this directly serve the one thing? Or does it serve a different
thing that's also nice to have?

Good products serve many use cases. Great products serve one primary experience brilliantly,
and everything else is in service of that. "It does a lot of things" is not a vision.

### Step 4: The third cut — the user's attention budget

Every element in a design costs attention. The user has a limited attention budget. Attention
spent on secondary elements is borrowed from the primary experience.

For each remaining item: how much of the user's attention does this require? Is that attention
well spent? What would the user experience if this weren't here — would they miss it, or would
the primary experience become clearer?

### Step 5: Unify through a deeper concept

The goal is not to end up with fewer things. The goal is to end up with one concept that
*contains* what the five used to do.

Ask: is there a more fundamental concept that covers all of these? If features A, B, and C
all exist because users want to organize their data — could a single, powerful organization
system replace all three specific features?

This is the hardest part. It requires understanding the problem at a deeper level than the
solutions you've already built. But it's the move that produces elegance.

## The "Back of the Cabinet" Pass

After cutting, run the back-of-the-cabinet check on what remains:

For each element that survived:
- Is it done with the same care as the most prominent element?
- If someone with exquisite taste looked at the details — the loading state, the error message,
  the empty state, the micro-animation — would they see the same level of craft?
- Is the invisible architecture (code organization, data model, API design) as clean as the
  visible design?

The "back of the cabinet" is not about visible polish. It's about whether the people building
it had integrity throughout — whether they finished the thing they couldn't see because that's
the kind of people they are.

## Common Failure Modes

| Failure | What it looks like | The real problem |
|---------|-------------------|-----------------|
| Feature parity trap | Adding features because competitors have them | Competing on the wrong dimension |
| User-request accumulation | Product grows by fulfilling every request | No one asking "what does this add up to?" |
| Hidden complexity | One button that reveals a hundred options | Complexity moved, not removed |
| Premature simplification | Removing power to reduce visible surface area | Didn't understand the underlying problem |
| Design simplicity without product simplicity | Clean UI over bloated functionality | Aesthetic simplicity vs. conceptual simplicity |

## Reference: Jobs's Simplicity Decisions

**The original Mac (1984):** No fans (silent), no cursor keys (mouse only), no command
line. Each removal was a fight. Each was the right call for the experience.

**The original iMac (1998):** Translucent colored plastic, handle, no floppy drive.
The "no floppy drive" was the most controversial. It accelerated the shift to internet-based
file transfer. The missing feature accelerated the future.

**The iPhone (2007):** No physical keyboard. No stylus. No SD card slot. No replaceable
battery. Each absence was attacked. Each was right.

**The MacBook Air (2008):** Removed the optical drive. Removed all ports except one.
The audience laughed at the product announcement. Then they bought one.

The pattern: each simplicity decision was attacked as a limitation at launch and understood
as correct in retrospect. The test of a simplicity decision is not immediate reception —
it's whether the experience it creates proves worth the absence.
