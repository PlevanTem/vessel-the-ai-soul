# File Parser Guide — 文件解析参考

> soul-forge Phase 2 的详细执行参考。
> 描述不同文件格式的解析策略和文本提取质量规范。

---

## 格式解析速查表

| 扩展名 | 解析方式 | 质量 | 备注 |
|--------|----------|------|------|
| `.md` | Read 工具直接读取 | ⭐⭐⭐ | 最佳格式，保留结构 |
| `.txt` | Read 工具直接读取 | ⭐⭐⭐ | 干净，无格式干扰 |
| `.pdf` | Read 工具（Cursor 内置 PDF 解析） | ⭐⭐ | 可能有排版残留，需清理 |
| `.json` | Read 后按结构提取文本字段 | ⭐⭐ | 见下方 JSON 处理规则 |
| `.csv` | Read 后提取文本列 | ⭐⭐ | 适合批量推文/评论 |
| `.html` | Read 后剥离标签 | ⭐ | 优先让用户转为 .md |
| `.docx` | ❌ 无法直接读取 | — | 提示用户另存为 .txt |
| `.pages` | ❌ 无法直接读取 | — | 提示用户另存为 .txt |
| `.epub` | ❌ 无法直接读取 | — | 提示用户导出章节文本 |

## 常见输入格式的处理规范

### Markdown / 纯文本文章

直接读取。提取时：
- 保留标题层级（帮助识别论点结构）
- 保留列表（常包含框架性思维）
- 去掉 YAML frontmatter（不含内容价值）
- 去掉纯格式标记（`---` 分隔线等）

**标记来源：** `*(Source: filename.md)*`

---

### JSON 格式（社交媒体导出）

#### Twitter/X 数据导出（官方归档）

```json
// 官方导出格式：data/tweets.js 或 tweet.js
// 提取字段：
{
  "tweet": {
    "full_text": "...",      // ← 主要提取这个
    "created_at": "...",
    "retweet_count": "...",
    "favorite_count": "..."
  }
}
```

过滤规则：
- 跳过 `full_text` 以 `RT @` 开头的转推
- 跳过纯回复（以 `@username` 开头且无额外内容）
- 保留引用推文（含评论的 retweet）

#### 微信聊天记录导出（常见工具格式）

```json
// 提取字段：sender + content，忽略系统消息
{ "sender": "Me", "content": "..." }
```

过滤：跳过 `[图片]` `[语音]` `[视频]` 等无文本消息

#### 笔记应用导出（Notion、Obsidian 等）

Notion 通常导出为 `.md`，Obsidian 本身就是 `.md`——直接读取。

---

### PDF 文档

Cursor Read 工具支持 PDF 文本提取。常见问题：
- **扫描版 PDF**：无法提取文字，提示用户使用 OCR 工具处理
- **排版残留**：页眉页脚、页码行可能混入正文，需识别并忽略
- **列布局**：双栏 PDF 文本顺序可能混乱，人工判断可读性

清理策略：
- 去掉连续出现的数字行（疑似页码）
- 去掉短于 20 字符且以数字结尾的行（疑似页眉/脚）
- 段落之间空行超过 3 行时合并

---

### 对话记录 / 访谈文字稿

常见格式：

```
主持人：你怎么看待...
受访者：我认为...
Q: ...
A: ...
[Name]: ...
```

提取规则：
- **只提取目标人物的发言**（受访者 / A / 目标名字标签下的内容）
- 保留问题作为上下文注释（标记为 `[Context: 问题内容]`）
- 口语化内容保留原貌（不要「修正」，语气词是声音特征的重要来源）

---

### 代码注释 / README / 技术文档

代码文件本身不用提取，关注：
- 文件顶部的 docstring / 模块注释（工程哲学）
- README 的「为什么做这个」段落
- 代码中的内联注释（尤其是解释决策的那种）
- Issue / PR 讨论记录中本人的发言

## 文本质量评估标准

提取完成后，对语料库做简单质量评分：

| 维度 | 检查项 |
|------|--------|
| **深度** | 是否有 500 词以上的连贯表达（文章/长篇日记）？ |
| **多样性** | 是否覆盖多种场景（正式/非正式，公开/私下）？ |
| **时间跨度** | 内容是否覆盖不同时期（有助于捕捉立场演化）？ |
| **第一人称** | 是否有足够多的第一人称表达（「我认为」「我做了」）？ |
| **原创性** | 是否主要为原创内容，而非转述/引用他人？ |

## 语料库组织格式

Phase 2 产出的统一语料库格式：

```markdown
## SOURCE: {文件名}
类型：{文章/日记/推文/访谈/...}
字数：{N}
时间：{YYYY-MM 或时间范围}

{提取的文本内容}

---

## SOURCE: {下一个文件名}
...
```

此格式在 Phase 3 DISTILL 中被直接引用，每条证据可溯源到具体文件。
